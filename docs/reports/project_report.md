# 项目报告：基于大模型的智能分诊与就医路径推荐系统

## 项目背景

医院导诊场景中，用户经常无法准确判断应该挂哪个科室，也难以区分普通症状和需要尽快就医的高风险症状。本项目以“智能分诊与就医路径推荐”为目标，使用大模型、规则工具、RAG 知识库和 LangGraph 工作流，将症状理解、风险分级、科室推荐、动态追问和就医路径生成串联成一个可演示的 Agent 系统。

项目定位是学习研究和工程展示，不替代医生诊断，也不直接提供治疗方案。

## 需求分析

系统需要支持以下需求：

- 用户能够用自然语言描述症状或报告问题。
- 系统需要识别症状、部位、严重程度、持续时间和就医意图。
- 系统需要优先识别胸痛、呼吸困难、意识异常、大量出血等高风险信号。
- 系统需要给出推荐科室、风险等级和推荐依据。
- 当信息不足时，系统应先追问关键问题，而不是直接给出确定结论。
- 系统需要给出院内位置、挂号流程和就医准备建议。
- 对超出能力范围或风险较高的问题，应建议人工转接或现场医护介入。
- 所有输出都需要保留医学安全边界，明确不能替代医生诊断。

## 系统设计

系统采用前后端分离和 Agent 工作流编排：

- 前端：`frontend/` 提供静态页面，用于输入症状、展示导诊结果、风险等级、科室、地图和模拟挂号信息。
- 后端：`backend/app.py` 提供 FastAPI 接口，负责接收请求、维护追问会话状态和返回结构化结果。
- Agent 流程：`agent_core/graph.py` 使用 LangGraph 编排导诊节点。
- 工具层：`tools/` 封装风险分级、科室推荐、RAG 检索、动态追问、人工转接等能力。
- 知识库：`data/docs/` 存放医学安全规则、科室说明、就诊流程和体检指标说明，`rag/` 负责构建和检索向量库。

## 系统模块设计

系统按照“入口层 -> 编排层 -> 能力层 -> 数据层 -> 展示层”组织，避免把所有判断都写在一个函数中，也方便后续替换模型、扩展知识库或接入真实医院系统。

| 模块 | 主要文件 | 设计说明 |
| --- | --- | --- |
| API 入口模块 | `backend/app.py` | 提供 `/chat`、`/robot/chat`、`/build` 等接口，负责请求参数校验、会话状态维护和结果字段规范化。 |
| 命令行演示模块 | `main.py`、`examples/demo_cases.json` | 提供无需 API Key 的轻量 demo，也支持通过 `--full-agent` 调用完整 LangGraph 流程，便于 GitHub 展示和面试演示。 |
| Agent 编排模块 | `agent_core/graph.py`、`agent_core/medical_graph.py` | 使用 LangGraph 定义导诊状态、节点和流转顺序，将风险分级、科室推荐、追问、RAG 和人工转接串联起来。 |
| 医疗工具模块 | `tools/` | 每个工具聚焦一个能力点，例如风险分级、科室路由、报告解读、院内地图、挂号指引和人工转接判断。 |
| 知识检索模块 | `rag/`、`data/docs/` | 管理本地文档切分、向量库构建和相似度检索，为回答提供可扩展的知识来源。 |
| 展示交互模块 | `frontend/` | 提供静态页面，展示用户输入、导诊结果、风险等级、推荐科室、院内位置和模拟挂号信息。 |

模块之间通过结构化字典传递结果，核心字段包括 `risk_level`、`recommended_department`、`need_followup`、`follow_up_questions`、`visit_guide` 和 `handoff`。这种设计让前端、命令行 demo 和机器人终端可以复用同一套导诊结果。

## 核心模块说明

### FastAPI 接口层

`backend/app.py` 提供 `/chat`、`/robot/chat`、`/build` 和 `/tools` 接口。其中 `/chat` 是主要入口，负责处理用户问题、合并追问信息、调用导诊图并规范化返回字段。

### LangGraph 流程层

`agent_core/graph.py` 定义 `MedicalAgentState`、节点函数和边关系，是当前核心业务流程。节点包括业务边界判断、风险分级、症状分析、科室推荐、院内位置、挂号流程、动态追问、RAG 检索、人工转接和最终回答生成。

### 导诊封装层

`agent_core/medical_graph.py` 提供 `run_medical_graph()`，负责初始化图状态、调用 LangGraph，并补齐前端需要的标准字段。

### 工具层

`tools/` 目录包含多个可复用工具：

- `risk_triage_tool.py`：识别高风险症状并输出风险等级。
- `department_router_tool.py`：根据症状规则推荐科室。
- `qwen_followup_tool.py`：生成动态追问问题。
- `medical_rag_tool.py`：检索本地医学知识库。
- `hospital_map_tool.py`：返回科室位置和院内路径。
- `registration_guide_tool.py`：生成挂号和就医流程。
- `lab_report_tool.py`：处理报告指标解释相关问题。
- `human_handoff_tool.py`：判断是否需要人工转接。

### RAG 知识库

`rag/build_vector_db.py` 读取 `data/docs/` 中的文本资料，切分后写入 ChromaDB。导诊过程中可通过 RAG 检索提供参考依据，提高回答的可解释性。

## 流程图文字版

```text
用户输入症状
  -> FastAPI /chat 接收请求
  -> 初始化 MedicalAgentState
  -> 业务边界判断
  -> 风险等级判断
  -> 症状与意图分析
  -> 科室推荐
  -> 科室位置查询
  -> 挂号流程生成
  -> 是否需要动态追问？
      -> 是：返回追问问题，等待用户补充
      -> 否：进入 RAG 检索
  -> 人工转接判断
  -> 生成最终导诊建议
  -> 返回风险等级、科室、路径、依据和免责声明
```

## 技术实现

- 使用 LangGraph 将导诊拆成多个节点，降低单次大模型调用承担的逻辑复杂度。
- 使用规则工具优先处理高风险信号和常见科室路由，保证基础能力稳定。
- 使用 OpenAI 兼容接口调用 Qwen 等大模型，完成语义理解、症状分析和追问生成。
- 使用 ChromaDB 和 sentence-transformers 构建本地知识库检索能力。
- 在完整 Agent 返回中保留 `sources` 和 `source_details` 字段，用于展示 RAG 参考来源，提升回答可解释性。
- 使用 FastAPI 暴露结构化接口，方便前端、机器人终端或命令行 demo 调用。
- 使用前端静态页面演示导诊结果、科室地图、医生列表和模拟挂号流程。
- 提供 Dockerfile 和 docker-compose.yml，支持容器化启动后端演示服务。
- 提供 `eval/triage_eval_cases.json` 和 `scripts/evaluate_demo_cases.py`，对轻量导诊 demo 做可复现评测。

## 项目不足

- 当前医学规则和知识库规模有限，不能覆盖复杂临床场景。
- OCR 报告识别尚未完成，报告解读主要依赖文本输入。
- RAG 知识库仍是轻量演示版本，缺少严格来源标注和医学审核流程。
- 当前评测主要覆盖轻量 demo，完整 LangGraph + LLM + RAG 流程仍需要独立评测集。
- 前端页面仍偏演示性质，距离真实医院业务系统还有差距。
- 部分历史文件和本地生成产物较多，开源时需要通过 `.gitignore` 控制提交范围。

## 后续优化

- 接入 OCR，支持检验报告、体检报告图片识别。
- 扩展医学知识库，加入可追溯来源和 RAG 引用展示。
- 扩展单元测试，继续覆盖更多 LangGraph 节点、工具函数和接口回归场景。
- 在已有 Dockerfile 和 docker-compose 的基础上补充线上部署脚本。
- 优化前端多轮会话体验和移动端适配。
- 增加管理端，用于维护科室规则、知识库文档和院内地图数据。
- 引入更严格的医学安全评估和日志审计机制。
